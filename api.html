FUNDAMENTALS · 01

APIs & REST

An API is how two programs talk to each other. Almost every app you use is constantly calling APIs — to log you in, load your feed, show the weather, take a payment. This guide explains what they are and how to use them.

🍽️
The restaurant analogy

You (the app) don't walk into the kitchen. You give your order to a waiter (the API), who takes it to the kitchen (the server) and brings back your food (the data). The menu is the list of things you're allowed to ask for.

This guide builds on JavaScript (you'll use fetch and async/await). By the end you'll build a GitHub user finder that calls a real REST API.

02

What is an API?

API = Application Programming Interface. It's a set of rules that lets one piece of software request data or actions from another — without knowing how that other software works inside.

📤

You send a request

"Give me the weather for Madurai." A message goes over the internet to a server.

📥

It sends a response

The server replies with data — usually JSON — and your app shows it.

📘
The key idea: a contract

An API is a promise: "send me a request shaped like this, and I'll send back data shaped like that." You don't need the server's code — just its rules (its documentation).

Web APIs (the kind here) work over HTTP — the same protocol your browser uses for web pages. Real examples: OpenWeatherMap (weather), the GitHub API (repos & users), Stripe/Razorpay (payments), Google Maps (locations).

03

REST principles

REST is the most common style for designing web APIs. It's not a technology — it's a set of conventions that make APIs predictable. An API that follows them is called RESTful.

✓
Everything is a resource — users, posts, products. Each has its own URL.
✓
Use HTTP methods — GET to read, POST to create, etc. (next section).
✓
Stateless — each request stands alone; the server doesn't remember the last one. You send everything it needs each time (like a token).
✓
Data as JSON — requests and responses carry JSON.
✓
Status codes — every response says how it went (200 ok, 404 not found…).
💡
In one sentence

A RESTful API exposes resources at URLs, and you act on them with HTTP methods, exchanging JSON. Learn that pattern once and almost every API feels familiar.

04

Endpoints & resources

An endpoint is a URL you can call. It's made of a base URL plus a path that names a resource. Resources are nouns (things), not verbs.

an endpoint, broken downurl
https://api.github.com/users/torvalds
└──────┬───────────┘ └──┬──┘ └───┬───┘
   base URL          resource    item
                   (collection)  (one user)
EndpointMeans
/usersThe whole collection of users
/users/torvaldsOne specific user (an item)
/users/torvalds/reposThat user's repositories (a sub-collection)
🚫
RESTful URLs use nouns

Good: GET /users/5. Avoid putting the action in the URL like /getUser?id=5 or /deleteUser — the method (GET, DELETE) already says the action.

05

HTTP methods → CRUD

The same URL can do different things depending on the method (also called the verb). The four basic data operations — Create, Read, Update, Delete (CRUD) — map directly onto HTTP methods.

MethodCRUDDoesExample
GETReadFetch data (never changes anything)GET /posts
POSTCreateAdd a new item (data in the body)POST /posts
PUTUpdateReplace an existing itemPUT /posts/1
PATCHUpdateChange part of an itemPATCH /posts/1
DELETEDeleteRemove an itemDELETE /posts/1

Here's all four against the free practice API jsonplaceholder.typicode.com:

crud.jsjavascript
const BASE = "https://jsonplaceholder.typicode.com";

// READ — get all posts
await fetch(`${BASE}/posts`);

// READ — get one post
await fetch(`${BASE}/posts/1`);

// CREATE — add a post (data goes in the body)
await fetch(`${BASE}/posts`, {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ title: "Hello", body: "My first post" })
});

// UPDATE — change a post
await fetch(`${BASE}/posts/1`, {
  method: "PUT",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ title: "Edited" })
});

// DELETE — remove a post
await fetch(`${BASE}/posts/1`, { method: "DELETE" });
⚠️
GET is "safe", the others change data

GET only reads — calling it twice is harmless. POST/PUT/DELETE change server data, so they need a method, usually headers, and (for POST/PUT) a body.

06

Status codes

Every response comes with a 3-digit status code telling you how it went. The first digit is the category.

RangeMeaningCommon ones
2xx✅ Success200 OK · 201 Created
3xx↪️ Redirect301 Moved · 304 Not Modified
4xx🙋 You messed up400 Bad Request · 401 Unauthorized · 403 Forbidden · 404 Not Found
5xx🔥 Server messed up500 Server Error · 503 Unavailable

With fetch, check response.ok (true for any 2xx) or response.status:

status.jsjavascript
const res = await fetch(url);

if (!res.ok) {                 // any 4xx or 5xx
  console.log("Failed:", res.status);   // e.g. 404
  return;
}
const data = await res.json();   // only read the body when it succeeded
💡
Go deeper

There's a whole guide on these — see HTTP status codes for the full list and when each one applies.

DATA · 07

Anatomy of a request

A request is more than a URL. It has four parts — and you'll set each of them when you call an API.

1
Method — GET, POST, PUT, DELETE (what to do).
2
URL — which resource, plus any params (which one).
3
Headers — extra info like Content-Type: application/json or an Authorization token.
4
Body — the data you're sending (POST/PUT only), as JSON.

🔗 Path params vs query params

Two ways to pass info in the URL:

paramsurl
/users/42            ← PATH param: WHICH resource (user 42)

/search?q=react&page=2   ← QUERY params: options after the "?"
                            (search for "react", page 2)
  • Path param (/users/42) identifies a specific resource.
  • Query params (?q=react&page=2) tweak the request — search terms, filters, paging. They start with ? and join with &.
DATA · 08

JSON — the data format

JSON (JavaScript Object Notation) is how APIs send data. It looks just like JavaScript objects and arrays — easy for humans to read and for code to parse.

example.jsonjson
{
  "name": "Linus Torvalds",
  "public_repos": 6,
  "followers": 220000,
  "languages": ["C", "Shell"],
  "company": null
}

JSON is text. JavaScript converts between text and real objects with two functions:

json.jsjavascript
// object → JSON text  (to SEND it in a request body)
JSON.stringify({ name: "Ada" });   // '{"name":"Ada"}'

// JSON text → object  (to USE what an API sent back)
JSON.parse('{"name":"Ada"}');      // { name: "Ada" }

// with fetch, response.json() does the parsing for you:
const data = await res.json();
🚫
JSON is strict

Keys must be in double quotes, no trailing commas, and no comments. { name: "Ada" } is valid JS but not valid JSON — it must be { "name": "Ada" }.

DATA · 09

Calling an API

In the browser, the built-in fetch is all you need. Here's the full pattern, with error handling — this is exactly what the mini project uses.

call.jsjavascript
async function getUser(username) {
  try {
    const res = await fetch(`https://api.github.com/users/${username}`);

    if (!res.ok) throw new Error("User not found");   // handle 404 etc.

    const data = await res.json();
    console.log(data.name, data.public_repos);
  } catch (err) {
    console.log("Error:", err.message);
  }
}
getUser("torvalds");

🧰 Axios & Postman

Two tools you'll meet:

  • Axios — a popular library (used a lot in React/Node) that's a little shorter than fetch and parses JSON automatically: const { data } = await axios.get(url).
  • Postman (or the VS Code REST Client) — apps to try an API by hand before writing code: set the method, URL, headers and body, hit Send, see the response. Great for exploring.
🔑
CORS & secret keys

Some APIs block browser calls (a security rule called CORS) or need a secret key. Those calls belong on a backend server, not in front-end JS — you'll build one in the Node & Express guides.

⚙️ JavaScript

script.jsjavascript
async function getUser() {
  const username = document.getElementById("username").value;
  const result = document.getElementById("result");

  if (username === "") {
    result.innerHTML = "<p>Enter a username 👆</p>";
    return;
  }

  result.innerHTML = "<p>Loading... ⏳</p>";

  try {
    // the endpoint: base URL + /users/ + the path param
    const res = await fetch(`https://api.github.com/users/${username}`);

    if (!res.ok) {                  // 404 = no such user
      result.innerHTML = "<p>User not found ❌</p>";
      return;
    }

    const user = await res.json();  // parse the JSON response

    result.innerHTML = `
      <img src="${user.avatar_url}" alt="${user.login}" />
      <h2>${user.name || user.login}</h2>
      <p>${user.bio || ""}</p>
      <div class="stats">
        <span>📦 ${user.public_repos} repos</span>
        <span>👥 ${user.followers} followers</span>
      </div>
    `;
  } catch (err) {
    result.innerHTML = "<p>Something went wrong ❌</p>";
  }
}
✓
The username becomes a path param in the endpoint URL.
✓
!res.ok catches a 404 before trying to read the body.
✓
user.name || user.login falls back gracefully when a field is empty.
