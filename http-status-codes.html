FUNDAMENTALS · 01

HTTP status codes

Every HTTP response carries a 3-digit number that summarises what happened — 200 (great), 404 (not found), 500 (server broke). Learn to read them and you can debug almost any web issue at a glance.

🚦
Think traffic lights

Status codes are the server's way of signalling outcome without words. The first digit is the colour of the light — it tells you the whole category before you read the rest.

This pairs with the Networking and APIs & REST guides — every response you saw there had one of these codes.

02

What is a status code?

It's a number the server puts at the top of every response, with a short text reason next to it. You usually never see it — the browser handles it — but as a developer you check it constantly.

a response status linehttp
HTTP/1.1 200 OK
              │   └─ reason phrase (for humans)
              └───── status code (for machines)

In JavaScript, fetch gives you the code on the response object:

check.jsjavascript
const res = await fetch(url);

res.status   // 200, 404, 500…  (the number)
res.ok       // true for any 2xx, false otherwise

if (!res.ok) console.log("Request failed:", res.status);
03

The five categories

The first digit tells you everything you need at a glance. There are five families.

RangeFamilyMeaning
1xxInformational"Hold on, still processing." Rare in everyday work.
2xx✅ SuccessIt worked.
3xx↪️ Redirection"Look somewhere else for this."
4xx🙋 Client errorYour request was wrong (bad data, not allowed, not found).
5xx🔥 Server errorThe request was fine, but the server failed.
💡
Remember the split

4xx = your fault (the client). 5xx = their fault (the server). That one distinction solves most "who do I blame?" debugging questions.

04

2xx — success

The request was received, understood, and accepted.

CodeNameWhen
200OKThe standard success — a GET returned data, an update worked.
201CreatedA POST made a new resource (a sign-up, a new post).
204No ContentSuccess, but nothing to send back — common after a DELETE.
05

3xx — redirects

"What you want is at a different URL" — or "you already have the latest copy."

CodeNameWhen
301Moved PermanentlyThis URL has a new home forever (browsers/SEO remember it).
302Found (temporary)Go here for now, but keep using the original URL.
304Not ModifiedYour cached copy is still fresh — saves re-downloading. Very fast.
📘
Redirects are automatic

The browser follows 3xx responses for you — that's why typing http:// often lands you on the https:// version.

06

4xx — client errors

Something about your request was wrong. The most common codes you'll meet and fix.

CodeNameWhen
400Bad RequestMalformed data — a missing field, invalid JSON.
401UnauthorizedNot logged in / no valid token. (Really means "unauthenticated.")
403ForbiddenLogged in, but not allowed to do this.
404Not FoundNo resource at that URL — the famous one.
409ConflictClashes with current state — e.g. that email is already registered.
422UnprocessableValid format but fails validation rules.
429Too Many RequestsYou've been rate-limited — slow down.
💡
401 vs 403

401 = "I don't know who you are" (log in). 403 = "I know who you are, and you can't." A subtle but important difference when building auth — see the Authentication guide.

07

5xx — server errors

Your request was fine; the server failed to handle it. These are the server's problem to fix.

CodeNameWhen
500Internal Server ErrorAn unhandled crash in the server code — the catch-all.
502Bad GatewayA server in front got a bad reply from one behind it.
503Service UnavailableOverloaded or down for maintenance — try again later.
504Gateway TimeoutAn upstream server took too long to respond.
⚠️
A 500 usually means "read the logs"

When your own API throws a 500, the real cause is in the server's console — an exception, a bad database query. The status is just the symptom.

08

Choosing the right code

When you build an API (with Express), you decide the status of each response. Returning the right one makes your API honest and easy to consume.

routes.js (Express)javascript
app.post("/users", (req, res) => {
  if (!req.body.email) {
    return res.status(400).json({ error: "Email is required" });   // bad input
  }
  if (emailTaken(req.body.email)) {
    return res.status(409).json({ error: "Email already exists" }); // conflict
  }

  const user = createUser(req.body);
  res.status(201).json(user);   // created!
});

app.get("/users/:id", (req, res) => {
  const user = findUser(req.params.id);
  if (!user) return res.status(404).json({ error: "Not found" });   // missing
  res.status(200).json(user);   // ok (200 is the default)
});
✓
Created something? 201, not 200.
✓
Bad input from the user? 400 (or 422 for validation).
✓
Can't find it? 404. Not allowed? 401/403.
✓
Never return 200 with an error message inside — the code is the signal.
MINI PROJECT

Status-code cheat sheet

Build a searchable one-page reference of the codes you just learned — type to filter, and keep it bookmarked. It's a neat exercise in arrays, map and a simple search filter.

🎯
What you'll build

A grid of code cards, colour-coded by family, with a search box that filters as you type (by number or name).

🧩 HTML

index.htmlhtml
<div class="sheet">
  <h1>HTTP Status Codes</h1>
  <input type="text" id="search" placeholder="Filter… e.g. 404 or 'not found'" />
  <div id="grid" class="grid"></div>
</div>

<script src="script.js"></script>

⚙️ JavaScript

script.jsjavascript
const codes = [
  { num: 200, name: "OK",            color: "#10b981" },
  { num: 201, name: "Created",       color: "#10b981" },
  { num: 301, name: "Moved",         color: "#3b82f6" },
  { num: 304, name: "Not Modified",  color: "#3b82f6" },
  { num: 400, name: "Bad Request",   color: "#f59e0b" },
  { num: 401, name: "Unauthorized",  color: "#f59e0b" },
  { num: 403, name: "Forbidden",     color: "#f59e0b" },
  { num: 404, name: "Not Found",     color: "#f59e0b" },
  { num: 500, name: "Server Error",  color: "#ef4444" },
  { num: 503, name: "Unavailable",   color: "#ef4444" },
];

const grid = document.getElementById("grid");
const search = document.getElementById("search");

function render(list) {
  grid.innerHTML = list
    .map(
      (c) => `
      <div class="code-card" style="--c:${c.color}">
        <div class="num">${c.num}</div>
        <div class="name">${c.name}</div>
      </div>`
    )
    .join("");
}

// filter as the user types (by number OR name)
search.addEventListener("input", () => {
  const q = search.value.toLowerCase();
  render(
    codes.filter(
      (c) => String(c.num).includes(q) || c.name.toLowerCase().includes(q)
    )
  );
});

render(codes);   // show everything on load
✓
The data lives in an array of objects — easy to extend with more codes.
✓
map() turns the data into cards; filter() powers the search.
✓
A CSS variable (--c) set inline colour-codes each card by family.
