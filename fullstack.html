02 · FRONTEND

Create your React project

The frontend is everything the user sees and clicks — buttons, forms, lists. We'll build it with React, and we'll create the project using a tool called Vite. Let's understand both before running any commands.

⚛️ Why React?

📘
What is React?

React is a JavaScript library for building user interfaces out of small, reusable pieces called components. Instead of one giant HTML file, you build a Button, a NoteCard, a Navbar — and snap them together like LEGO bricks.

🧱
Real-world analogy

Building a webpage with plain HTML is like writing a whole essay by hand every time. React is like having reusable paragraphs (components) — write a "NoteCard" once, then reuse it for 100 notes. Change it in one place, and all 100 update.

⚡ Why Vite (and not Create React App)?

Vite (pronounced "veet", French for "quick") is a tool that creates and runs your React project. The older tool, Create React App (CRA), is now effectively retired and is much slower. Here's the honest comparison:

 Vite ⚡ (recommended)Create React App 🐢 (legacy)
Startup speedAlmost instantSlow, gets slower as app grows
Hot reloadLightning fastNoticeably laggy
MaintenanceActively developedNo longer recommended
ConfigTiny, simpleHidden, harder to customise
Commandnpm create vite@latestnpx create-react-app
💡
Verdict

Use Vite. It's faster, simpler, and what the React team itself now points beginners toward. We'll use it for the rest of this guide.

⌨️ Step 1 — Create the project

Open your terminal, navigate to where you keep projects (e.g. your Desktop), and run:

Terminalbash
npm create vite@latest frontend

Breaking this command down word by word:

  • npm create — tells npm to scaffold (generate) a new project from a template.
  • vite@latest — use the newest version of the Vite project generator.
  • frontend — the folder name to create. We deliberately call it frontend because later we'll have a separate backend folder.

Vite will ask you a few questions — here's what to pick

QuestionChooseWhy
Select a frameworkReactWe're building a React app.
Select a variantJavaScriptWe're using plain JavaScript, not TypeScript — simpler for beginners.
❓
Why JavaScript and not TypeScript?

TypeScript adds type-checking, which is powerful but adds extra rules to learn. As a beginner, plain JavaScript lets you focus on understanding React and the full-stack flow first. You can add TypeScript later.

📂 Step 2 — Go into the folder

Terminalbash
cd frontend

cd means "change directory". Your terminal is like a file explorer with no windows — cd frontend walks into the new folder so the next commands run in the right place.

📦 Step 3 — Install dependencies

Terminalbash
npm install

Vite created a list of required libraries (in package.json) but didn't download them yet. npm install reads that list and downloads everything into a folder called node_modules. This can take 10–30 seconds and only needs to be done once (and again whenever you add a new package).

🤔 What is "localhost" and why 5173?

📘
localhost = your own computer

localhost is a special address that always means "this very computer". The number after the colon (5173) is the port — like a specific door into your computer. Vite uses door 5173 by default. So http://localhost:5173 means "open the website my computer is serving on door 5173".

🔥 What is Hot Reload (HMR)?

While npm run dev is running, edit any file and hit save. The browser updates instantly without a full refresh — and without losing your place. This is called Hot Module Replacement. Try it: open src/App.jsx, change some text, save, and watch the browser update in real time.

💡
Leave it running

Keep this terminal open while you work. To stop the server, click the terminal and press Ctrl + C. To start again, run npm run dev.

🗂️ Understanding every generated file

Vite generated a small, clean folder structure. Here it is, followed by an explanation of every single file:

frontend/tree
frontend/
├── node_modules/        # Installed libraries (auto-generated, never edit)
├── public/              # Static files served as-is (favicon, images)
│   └── vite.svg
├── src/                 # YOUR code lives here
│   ├── assets/          # Images/logos used inside components
│   │   └── react.svg
│   ├── App.css          # Styles for the App component
│   ├── App.jsx          # The main/root React component
│   ├── index.css        # Global styles for the whole app
│   └── main.jsx         # The entry point — boots React into the page
├── .gitignore           # Tells Git which files to ignore
├── index.html           # The single HTML page React mounts into
├── package.json         # Project info + scripts + dependency list
├── package-lock.json    # Exact versions of every installed package
└── vite.config.js       # Vite's configuration file

File-by-file explanation

  • index.html — Surprisingly, a React app has just one HTML file. It contains a single empty <div id="root"></div>. React injects your entire UI inside that div.
  • src/main.jsx — The entry point. It finds #root in index.html and tells React to render your <App /> there. This file runs first.
  • src/App.jsx — The root component. This is where you'll start building your UI. .jsx means "JavaScript + HTML-like syntax".
  • src/App.css / src/index.css — Stylesheets. index.css is global; App.css is scoped to App by convention.
  • src/assets/ — Put images/icons here that are imported by components.
  • public/ — Files here are copied to the site root untouched. Good for a favicon or robots.txt.
  • package.json — The project's "ID card". Lists its name, the scripts you can run (like dev), and its dependencies.
  • package-lock.json — Auto-generated. Records the exact version of every package so the app installs identically on any machine. Don't edit it by hand.
  • vite.config.js — Vite's settings. For now it just enables the React plugin; later you can add a proxy or change the port here.
  • node_modules/ — Thousands of downloaded library files. Huge, auto-generated, and never committed to Git.
  • .gitignore — A list of files/folders Git should ignore (like node_modules).
🚫
Common mistakes

• Running npm run dev from the wrong folder (you forgot cd frontend). The error: Could not read package.json.

• Editing files inside node_modules — never do this; your changes get wiped on the next install.

• Forgetting to run npm install before npm run dev → error: vite: command not found.

04 · BACKEND

Create the backend project

The backend is the "kitchen" of your app — the user never sees it, but it does the real work: talking to the database, applying rules, and sending data back. We build it as a completely separate project from the frontend.

❓
Why a separate folder/project?

Frontend and backend are two different programs that run independently — possibly even on different servers. Keeping them in separate folders (each with its own package.json) keeps their dependencies clean and lets you deploy them separately to Vercel. Mixing them causes confusing version clashes.

📁 Step 1 — Create & enter the folder

Go up and out of the frontend folder first, so backend sits next to it, not inside it:

  • cd backend — step into it. All following commands run here.

05 · BACKEND

Create the Express server

Time to write the heart of the backend: server.js. We'll build it up and explain every single line. Create a new file called server.js in the backend folder.

🧩 The complete starter server

backend/server.jsjavascript
// 1. Import the packages we installed
import express from "express";   // the web framework
import cors from "cors";         // lets the frontend call us
import dotenv from "dotenv";     // loads secrets from .env

// 2. Load variables from the .env file into process.env
dotenv.config();

// 3. Create the Express application
const app = express();

// 4. Decide which port to run on (from .env, or 5000 as a fallback)
const PORT = process.env.PORT || 5000;

// 5. Middleware — code that runs on EVERY incoming request
app.use(cors());            // allow cross-origin requests from the frontend
app.use(express.json());    // automatically parse JSON request bodies

// 6. A simple test route so we can confirm the server is alive
app.get("/", (req, res) => {
  res.send("API is running ✅");
});

// 7. Start listening for requests
app.listen(PORT, () => {
  console.log(`Server running on http://localhost:${PORT}`);
});

🔍 Line-by-line explanation

  • Imports — we pull in express, cors and dotenv, the packages we installed with npm.
  • dotenv.config() — reads your .env file and copies its values into process.env so you can use them. Must run before you read any env variable.
  • const app = express() — creates your application object. Everything (routes, middleware) attaches to app.
  • const PORT = process.env.PORT || 5000 — uses the port from .env if it exists, otherwise defaults to 5000. The || means "or".
  • app.use(cors()) — middleware that adds the headers browsers need to allow your React app to call this server (full deep-dive next section).
  • app.use(express.json()) — middleware that reads incoming JSON and puts it on req.body. Without it, req.body would be undefined.
  • app.get("/", ...) — defines a route. When someone visits /, the function runs. req = the incoming request, res = your response. res.send(...) sends text back.
  • app.listen(PORT, ...) — actually starts the server and keeps it running, listening for requests on the chosen port. The callback prints a confirmation to the terminal.
📘
What is "middleware"?

Middleware is simply a function that runs between receiving a request and sending a response. Think of airport security: every passenger (request) passes through the same checkpoints (middleware) before reaching the gate (your route). cors() and express.json() are two such checkpoints.

06 · THE BIG ONE

Understanding CORS (in depth)

CORS confuses almost every beginner. By the end of this section you'll understand exactly what it is, why browsers enforce it, why Postman ignores it, and how to configure it safely. We'll use diagrams throughout.

🌍 First: what is an "origin"?

An origin is the combination of three things: protocol + domain + port. If any of those three differ between two URLs, they are different origins.

Frontend (React / Vite)http://localhost:5173
≠
Backend (Express)http://localhost:5000

Same protocol (http), same domain (localhost) — but the port differs (5173 vs 5000). That alone makes them different origins, and that's why CORS comes into play even on your own laptop.

🔒 The Same-Origin Policy

📘
What is the Same-Origin Policy (SOP)?

A built-in browser security rule. By default, JavaScript running on one origin is not allowed to read responses from a different origin. So your React code at :5173 is blocked from reading data at :5000 — unless the backend explicitly says "I allow this".

❓
Why would browsers block this? It seems annoying!

Imagine you're logged into your bank in one tab. You open a shady website in another tab. Without the Same-Origin Policy, that shady site's JavaScript could quietly call your bank's API using your logged-in session and steal your data. SOP stops that. CORS is the safe, controlled way for a server to say "these specific origins are allowed."

🚫 Why React fails but Postman works

🧭 Without CORS configured
🌐Browserlocalhost:5173
⛔SOP blocks the response
🚀Expresslocalhost:5000
🧪 Postman (not a browser)
📮Postmandesktop app
✅No SOP — request succeeds
🚀Expresslocalhost:5000
💡
The key insight

CORS is enforced only by browsers. Postman, curl, and your backend itself are not browsers, so they never apply the Same-Origin Policy. That's why "it works in Postman but fails in React" — the server is fine; the browser is the one blocking it until CORS headers are present.

✈️ Preflight requests & the OPTIONS method

For certain requests (like a POST with JSON, or anything with custom headers), the browser sends a quiet "permission check" first, before the real request. This is called a preflight request, and it uses the HTTP OPTIONS method.

✈️ A preflight handshake
🌐Browser1 · OPTIONS ?
➡️"May I POST here?"
🚀Server2 · Yes, allowed
➡️3 · real POST sent
🍃DB4 · data saved

If your server doesn't answer the OPTIONS preflight with the right headers, the browser cancels the real request and you see a CORS error. The good news: the cors package handles preflight automatically.

🛡️ Using the cors middleware

The simplest version (allows everyone)

server.jsjavascript
import cors from "cors";

// Allow requests from ANY origin (fine for early local development)
app.use(cors());
⚠️
Good for learning, risky for production

cors() with no options sends Access-Control-Allow-Origin: * — meaning any website can call your API. That's convenient while learning but in production you should restrict it to your own frontend.

The production-ready version (allow only your frontend)

server.jsjavascript
import cors from "cors";

const allowedOrigins = [
  "http://localhost:5173",            // local React (Vite) dev server
  "https://your-app.vercel.app"       // your deployed frontend
];

app.use(
  cors({
    origin: allowedOrigins,           // who is allowed to call us
    methods: ["GET", "POST", "PUT", "DELETE"], // which HTTP verbs are allowed
    allowedHeaders: ["Content-Type", "Authorization"], // which request headers are allowed
    credentials: true                 // allow cookies / auth headers to be sent
  })
);

Every option, explained

OptionWhat it controls
originWhich websites may call your API. A single string, an array, or true (reflect any). This becomes the Access-Control-Allow-Origin header.
methodsWhich HTTP verbs (GET, POST, PUT, DELETE, PATCH) the browser is allowed to use.
allowedHeadersWhich custom request headers the frontend may send (e.g. Content-Type, Authorization).
credentialsIf true, the browser may send cookies / auth headers. Note: you cannot use credentials: true together with origin: "*".

⚖️ Good vs. bad examples

✅

Good (production)

Explicit list of trusted origins, only the methods you use, credentials handled deliberately.

❌

Bad (production)

app.use(cors()) left wide open on a real app, exposing your API to every website on the internet.

🔐 Security best practices

✓
In production, whitelist only your real frontend domain(s).
✓
Store allowed origins in environment variables, not hardcoded.
✓
Only enable credentials: true if you actually use cookies/auth.
✓
List only the HTTP methods your API really supports.

🐞 The error you'll inevitably see

Chrome DevTools Consoleerror
Access to XMLHttpRequest at 'http://localhost:5000/api/notes'
from origin 'http://localhost:5173' has been blocked by CORS policy:
No 'Access-Control-Allow-Origin' header is present on the requested resource.
🚫
How to fix this error — checklist

1. Did you actually call app.use(cors())? 2. Is it placed before your routes? 3. Is the backend server actually running? 4. In production, is your frontend's URL in the origin list (exact match, including https:// and no trailing slash)?

🎯
Summary

CORS is a browser safety mechanism. Your server uses the cors package to attach Access-Control-Allow-* headers that tell the browser "these origins are allowed." Use cors() while learning, switch to an explicit origin list in production.

07 · DATABASE

Set up MongoDB Atlas

Your app needs somewhere to store data permanently — notes, contact messages, users. We'll use MongoDB Atlas: a free, cloud-hosted MongoDB database. No installation needed; it lives online and your backend connects to it.

📘
What is MongoDB? What is Atlas?

MongoDB is a database that stores data as flexible JSON-like documents (instead of rigid tables). Atlas is MongoDB's official cloud service that hosts your database for you, with a generous free tier — perfect for learning and small projects.

🗄️
Analogy

A database is a smart filing cabinet. A collection is a drawer (e.g. "notes"). A document is a single folder inside that drawer (one note). MongoDB lets each folder hold flexible contents instead of forcing identical forms.

2️⃣ The Controller — controllers/noteController.js

The controller holds the logic for each action. Each function is an async handler receiving req (request) and res (response).

backend/controllers/noteController.jsjavascript
import Note from "../models/Note.js";

// GET /api/notes — read all notes (newest first)
export const getNotes = async (req, res) => {
  try {
    const notes = await Note.find().sort({ createdAt: -1 });
    res.status(200).json(notes);
  } catch (err) {
    res.status(500).json({ message: err.message });
  }
};

// POST /api/notes — create a note
export const createNote = async (req, res) => {
  try {
    const { title, content } = req.body;        // data sent by the frontend
    if (!title) {
      return res.status(400).json({ message: "Title is required" });
    }
    const newNote = await Note.create({ title, content });
    res.status(201).json(newNote);              // 201 = created
  } catch (err) {
    res.status(500).json({ message: err.message });
  }
};

// PUT /api/notes/:id — update a note
export const updateNote = async (req, res) => {
  try {
    const updated = await Note.findByIdAndUpdate(
      req.params.id,        // which note (from the URL)
      req.body,             // the new data
      { new: true }         // return the UPDATED document
    );
    if (!updated) return res.status(404).json({ message: "Note not found" });
    res.status(200).json(updated);
  } catch (err) {
    res.status(500).json({ message: err.message });
  }
};

// DELETE /api/notes/:id — delete a note
export const deleteNote = async (req, res) => {
  try {
    const deleted = await Note.findByIdAndDelete(req.params.id);
    if (!deleted) return res.status(404).json({ message: "Note not found" });
    res.status(200).json({ message: "Note deleted" });
  } catch (err) {
    res.status(500).json({ message: err.message });
  }
};

Key ideas explained

  • Note.find() — fetches all documents; .sort({ createdAt: -1 }) puts newest first.
  • req.body — the JSON the frontend sent. Available because of express.json() middleware.
  • req.params.id — the :id part of the URL, e.g. /api/notes/abc123.
  • res.status(201) — sets the HTTP status code (201 = "Created"). .json(...) sends JSON back.
  • { new: true } — tells Mongoose to return the note after updating, not before.
  • Each function is wrapped in try/catch so database errors return a clean 500 instead of crashing.

import { useState, useEffect } from "react";
import api from "../services/api";
import "./Notes.css";

export default function Notes() {
  // STATE — React re-renders the UI whenever these change
  const [notes, setNotes] = useState([]);          // the list of notes
  const [title, setTitle] = useState("");          // form: title input
  const [content, setContent] = useState("");      // form: content input
  const [editingId, setEditingId] = useState(null);// id of note being edited

  // READ — fetch all notes once, when the page first loads
  const fetchNotes = async () => {
    try {
      const res = await api.get("/api/notes");
      setNotes(res.data);                          // store them in state
    } catch (err) {
      console.error("Failed to load notes:", err.message);
    }
  };

  useEffect(() => {
    fetchNotes();                                  // run once on mount
  }, []);                                          // [] = "only the first time"

  // CREATE or UPDATE — depending on whether we're editing
  const handleSubmit = async (e) => {
    e.preventDefault();                            // stop the page reloading
    if (!title.trim()) return;                     // ignore empty titles

    try {
      if (editingId) {
        // UPDATE existing note
        await api.put(`/api/notes/${editingId}`, { title, content });
      } else {
        // CREATE new note
        await api.post("/api/notes", { title, content });
      }
      // reset the form and refresh the list
      setTitle("");
      setContent("");
      setEditingId(null);
      fetchNotes();
    } catch (err) {
      console.error("Save failed:", err.message);
    }
  };

  // Put a note's values back into the form to edit it
  const startEdit = (note) => {
    setEditingId(note._id);
    setTitle(note.title);
    setContent(note.content);
  };

  // DELETE a note by id
  const handleDelete = async (id) => {
    try {
      await api.delete(`/api/notes/${id}`);
      fetchNotes();
    } catch (err) {
      console.error("Delete failed:", err.message);
    }
  };

  return (
    <div className="notes-page">
      <h1>📝 My Sticky Notes</h1>

      {/* The form handles both creating and editing */}
      <form onSubmit={handleSubmit} className="note-form">
        <input
          placeholder="Title"
          value={title}
          onChange={(e) => setTitle(e.target.value)}
        />
        <textarea
          placeholder="Write your note..."
          value={content}
          onChange={(e) => setContent(e.target.value)}
        />
        <button type="submit">
          {editingId ? "Update Note" : "Add Note"}
        </button>
      </form>

      {/* The list of notes */}
      <div className="note-grid">
        {notes.map((note) => (
          <div className="note-card" key={note._id}>
            <h3>{note.title}</h3>
            <p>{note.content}</p>
            <div className="note-actions">
              <button onClick={() => startEdit(note)}>✏️ Edit</button>
              <button onClick={() => handleDelete(note._id)}>🗑️ Delete</button>
            </div>
          </div>
        ))}
      </div>
    </div>
  );
}

⚛️ Part B — The Frontend Form

The form keeps all four fields in one state object, validates them, and shows inline errors plus a success message.

frontend/src/pages/Contact.jsxjsx
import { useState } from "react";
import api from "../services/api";
import "./Contact.css";

const initialForm = { name: "", email: "", phone: "", message: "" };

export default function Contact() {
  const [form, setForm] = useState(initialForm);
  const [errors, setErrors] = useState({});
  const [success, setSuccess] = useState("");
  const [loading, setLoading] = useState(false);

  // Update one field by its "name" attribute
  const handleChange = (e) => {
    setForm({ ...form, [e.target.name]: e.target.value });
  };

  // Simple client-side validation; returns an errors object
  const validate = () => {
    const err = {};
    if (!form.name.trim()) err.name = "Name is required";
    if (!form.email.trim()) {
      err.email = "Email is required";
    } else if (!/^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(form.email)) {
      err.email = "Enter a valid email";
    }
    if (!/^[0-9]{7,15}$/.test(form.phone)) err.phone = "Enter a valid phone number";
    if (!form.message.trim()) err.message = "Message cannot be empty";
    return err;
  };

  const handleSubmit = async (e) => {
    e.preventDefault();
    setSuccess("");
    const foundErrors = validate();
    setErrors(foundErrors);
    if (Object.keys(foundErrors).length > 0) return; // stop if invalid

    try {
      setLoading(true);
      const res = await api.post("/api/contacts", form);
      setSuccess(res.data.message);   // "Thanks! Your message was received."
      setForm(initialForm);           // clear the form
    } catch (err) {
      setSuccess("");
      setErrors({ submit: err.response?.data?.message || "Something went wrong" });
    } finally {
      setLoading(false);
    }
  };

  return (
    <div className="contact-page">
      <h1>📨 Contact Us</h1>

      {success && <p className="success">✅ {success}</p>}
      {errors.submit && <p className="error">❌ {errors.submit}</p>}

      <form onSubmit={handleSubmit} noValidate>
        <input name="name" placeholder="Name"
               value={form.name} onChange={handleChange} />
        {errors.name && <span className="field-error">{errors.name}</span>}

        <input name="email" placeholder="Email"
               value={form.email} onChange={handleChange} />
        {errors.email && <span className="field-error">{errors.email}</span>}

        <input name="phone" placeholder="Phone"
               value={form.phone} onChange={handleChange} />
        {errors.phone && <span className="field-error">{errors.phone}</span>}

        <textarea name="message" placeholder="Your message"
               value={form.message} onChange={handleChange} />
        {errors.message && <span className="field-error">{errors.message}</span>}

        <button type="submit" disabled={loading}>
          {loading ? "Sending..." : "Send Message"}
        </button>
      </form>
    </div>
  );
}

Axios — talking to your backend

Axios is the library your React app uses to send HTTP requests to the backend. You've already used it; now let's understand it properly with examples for every method.

❓
Why Axios and not plain fetch?

The browser has a built-in fetch(), but Axios adds conveniences: it automatically converts JSON, lets you set a baseURL once, throws errors on bad status codes, and has cleaner syntax. For beginners it removes a lot of boilerplate.

📦 Installation

Terminal (frontend)bash
npm install axios

🔡 The four methods, with examples

axios examplesjavascript
import api from "../services/api"; // our pre-configured axios instance

// GET — read data
const res = await api.get("/api/notes");
console.log(res.data);            // the array of notes

// POST — create data (second argument is the body)
await api.post("/api/notes", { title: "Hello", content: "World" });

// PUT — update data by id
await api.put("/api/notes/123", { title: "Updated title" });

// DELETE — remove data by id
await api.delete("/api/notes/123");
📘
The golden rule: data is in res.data

Axios wraps the server's reply in a response object. The actual payload your backend sent is always on res.data — not res itself. Other useful fields: res.status (e.g. 200) and res.headers.

⏳ async / await explained

HTTP requests take time. await tells JavaScript "pause here until the response arrives, then continue" — without freezing the whole app. A function using await must be marked async.

error handling patternjavascript
async function loadNotes() {
  try {
    const res = await api.get("/api/notes");
    return res.data;                  // success path
  } catch (err) {
    // err.response exists when the server replied with an error status
    if (err.response) {
      console.error("Server said:", err.response.status, err.response.data);
    } else {
      console.error("Network/other error:", err.message);
    }
  }
}
💡
Always wrap requests in try/catch

If the backend is down or returns a 4xx/5xx, Axios throws. Catching it lets you show a friendly message instead of a blank screen. err.response contains the server's status and body; err.message covers network failures.

🚫
Common mistakes

• Forgetting await → you get a Promise object instead of the data.

• Reading res instead of res.data.

• Calling await in a function that isn't async → syntax error.

17 · REFERENCE

Common errors & how to fix them

Every developer hits these. Bookmark this section — when something breaks, find the error here and follow the fix. Click any item to expand it.

Cause: the package isn't installed, or the import path is wrong.

Fix: run npm install. For your own files, check the path and extension — with "type":"module" you must include .js, e.g. import x from "./config/db.js".

Cause: you visited a route the server doesn't define (e.g. opening the backend root when no / route exists).

Fix: add a route like app.get("/", ...), or visit the correct endpoint such as /api/notes. It's harmless if you just forgot the path.

Causes & fixes:

• bad auth → wrong username/password in MONGO_URI.

• ServerSelectionError / timeout → your IP isn't whitelisted in Atlas Network Access (add 0.0.0.0/0), or no internet.

• undefined URI → missing dotenv.config() or wrong .env location.

Cause: another process is already using port 5000 (often a server you forgot to stop).

Fix: stop the old terminal (Ctrl + C), or change PORT in .env. On Mac/Linux: lsof -i :5000 then kill -9 <PID>. On Windows: netstat -ano | findstr :5000 then taskkill /PID <PID> /F.

Cause: backend isn't sending CORS headers, or doesn't allow your frontend's origin.

Fix: add app.use(cors()) (before routes) for dev, or add your frontend URL to the origin list for production. See Section 06.

Cause: the backend isn't running, the URL is wrong, or it's a CORS issue.

Fix: confirm the backend is up, check VITE_API_URL, open the browser Network tab to see the failing request, and inspect err.response for the server's message.

Cause: an unhandled error in your backend code (often a database or null-property issue).

Fix: read the backend terminal — the stack trace points to the line. Ensure handlers use try/catch and that you validate req.body.

Cause: the URL/method doesn't match any route (typo, wrong method, missing /api prefix).

Fix: compare the request to your route definitions. GET /notes won't match app.use("/api/notes", ...) — it must be /api/notes.

Cause: Node.js isn't installed or isn't on your PATH, or the terminal was open before installing.

Fix: reinstall Node (LTS) keeping "Add to PATH" checked, then fully close and reopen the terminal.

Cause: missing env vars, missing export default app, or a build error.

Fix: open Vercel → Logs for the exact error. Add every env var from your local .env into the Vercel dashboard and redeploy.

    await Message.create({ role: "user", text: message });

    // 2. send the recent conversation to Gemini (so it has memory)
    const history = await Message.find().sort({ createdAt: 1 }).limit(20);

    const response = await fetch(`${GEMINI_URL}?key=${process.env.GEMINI_API_KEY}`, {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({
        contents: history.map((m) => ({
          // Gemini calls the AI side "model" instead of "assistant"
          role: m.role === "assistant" ? "model" : "user",
          parts: [{ text: m.text }],
        })),
      }),
    });

    const data = await response.json();
    const reply =
      data.candidates?.[0]?.content?.parts?.[0]?.text || "Sorry, I couldn't reply.";

    // 3. save the AI's reply and send it back
    const aiMessage = await Message.create({ role: "assistant", text: reply });
    res.json(aiMessage);
  } catch (err) {
    console.error(err);
    res.status(500).json({ error: "AI request failed" });
  }
};
💡
Using Claude or OpenAI instead?

Only this file changes. The idea is identical — send the conversation, read back the reply. e.g. OpenAI: ai.chat.completions.create({ model, messages }) → completion.choices[0].message.content; Anthropic: ai.messages.create(...) → completion.content[0].text.

import { useState, useEffect, useRef } from "react";
import axios from "axios";
import "./ChatWindow.css";

const API = "http://localhost:5000/api/chat";

export default function ChatWindow() {
  const [messages, setMessages] = useState([]);
  const [input, setInput] = useState("");
  const [loading, setLoading] = useState(false);
  const endRef = useRef(null);

  // load the saved conversation once
  useEffect(() => {
    axios.get(API).then((res) => setMessages(res.data));
  }, []);

  // auto-scroll to the newest message
  useEffect(() => {
    endRef.current?.scrollIntoView({ behavior: "smooth" });
  }, [messages, loading]);

  async function send() {
    const text = input.trim();
    if (!text) return;
    setInput("");
    setMessages((prev) => [...prev, { role: "user", text }]); // show instantly
    setLoading(true);
    try {
      const res = await axios.post(API, { message: text });
      setMessages((prev) => [...prev, res.data]);             // AI reply
    } catch {
      setMessages((prev) => [...prev, { role: "assistant", text: "⚠️ Something went wrong." }]);
    } finally {
      setLoading(false);
    }
  }

  return (
    <div className="chat">
      <header className="chat-head">
        <span className="chat-avatar">🤖</span>
        <div><strong>AI Assistant</strong><small>● online</small></div>
      </header>

      <div className="chat-body">
        {messages.map((m, i) => (
          <div key={i} className={`bubble ${m.role}`}>{m.text}</div>
        ))}
        {loading && (
          <div className="bubble assistant typing"><span></span><span></span><span></span></div>
        )}
        <div ref={endRef} />
      </div>

      <div className="chat-input">
        <input
          value={input}
          onChange={(e) => setInput(e.target.value)}
          onKeyDown={(e) => e.key === "Enter" && send()}
          placeholder="Ask me anything…"
        />
        <button onClick={send} aria-label="Send">➤</button>
      </div>
    </div>
  );
}
