Git & GitHub

Git is the tool that saves snapshots of your code over time, so you can undo mistakes, work on features in parallel, and collaborate without overwriting each other. GitHub is where you store those snapshots online. Every developer uses them, every day.

🎮
Save points for code

Git is like the save system in a game. Each commit is a save point you can return to. Branches let you explore a risky path, and if it goes wrong, you reload your last good save — your real work is never lost.

⚖️
Git ≠ GitHub

Git is the version-control program on your computer. GitHub (also GitLab, Bitbucket) is a website that hosts Git repositories so you can back them up and share. You can use Git with no GitHub at all.

02

What is version control?

Without it, you end up with project-final.zip, project-final-v2.zip, project-FINAL-really.zip. Version control replaces all that with a clean, searchable history of every change.

✓
History — see exactly what changed, when, and who changed it.
✓
Undo — roll any file (or the whole project) back to a previous state.
✓
Branching — try new ideas in isolation without breaking the working version.
✓
Collaboration — many people edit the same project and merge their work safely.
03

Setup & creating a repo

Install Git from git-scm.com, then set your name and email once (they get stamped on every commit).

one-time setupbash
git --version                      # check it's installed

git config --global user.name  "Your Name"
git config --global user.email "you@example.com"

A Git project is called a repository (repo). There are two ways to get one:

start a repobash
# A) turn an existing folder into a repo
cd my-project
git init

# B) copy an existing repo from GitHub onto your machine
git clone https://github.com/user/repo.git
📘
The hidden .git folder

git init creates a hidden .git folder — that's where Git stores your entire history. Delete it and the folder becomes a normal, un-tracked directory again.

04

Staging & committing

Saving a change in Git is a two-step move: stage the changes you want (git add), then commit them with a message (git commit). This lets you choose exactly what goes into each snapshot.

the three areastext
Working dir   ──git add──▶   Staging area   ──git commit──▶   Repository
(your edits)                 (ready to save)                  (saved history)
save a changebash
git add index.html         # stage one file
git add .                  # or stage everything changed

git commit -m "Add hero section"   # save a snapshot with a message
💡
Write good commit messages

Short, present-tense, and describing what the change does: "Add login form", "Fix navbar on mobile". Your future self (and teammates) will thank you when reading the history.

05

Seeing what's going on

Three commands you'll run constantly to understand the state of your repo.

CommandShows
git statusWhat's changed, staged, or untracked right now
git logThe history of commits (newest first)
git log --onelineThe same, one compact line per commit
git diffThe exact lines you've changed but not staged
git statusbash
$ git status
On branch main
Changes not staged for commit:
  modified:   index.html
Untracked files:
  style.css
💡
When in doubt, run git status

It's the most useful command in Git — it tells you what state you're in and usually hints at the next command to run.

06

Branches & merging

A branch is an independent line of work. You branch off main, build a feature in peace, then merge it back when it's ready — main stays stable the whole time.

branchingbash
git branch                     # list branches (* marks current)
git switch -c feature-login    # create AND switch to a new branch
                               # (older syntax: git checkout -b feature-login)

# ...do your work, add & commit on this branch...

git switch main                # go back to main
git merge feature-login        # bring the feature's commits into main

git branch -d feature-login    # delete the branch once merged
📘
Why branch?

So unfinished or experimental work never breaks the version everyone relies on. On teams, every feature or fix gets its own branch, reviewed via a Pull Request before merging.

07

Push & pull (remotes)

A remote is a copy of your repo hosted elsewhere — usually GitHub, nicknamed origin. Push sends your commits up; pull brings others' commits down.

remotesbash
# connect your local repo to a GitHub repo (done once)
git remote add origin https://github.com/you/repo.git

git push -u origin main    # first push (sets the upstream)
git push                   # every push after that

git pull                   # fetch + merge others' changes into yours
CommandDirection
git push⬆️ Local commits → GitHub
git pull⬇️ GitHub commits → Local
git clone⬇️ A whole repo → a new local copy
💡
Pull before you push

If teammates pushed while you worked, git pull first to bring their changes in — then push. This avoids most "rejected" push errors.

08

.gitignore

Some files should never be committed — huge dependency folders, secret keys, build output. List them in a .gitignore file and Git will pretend they don't exist.

.gitignoretext
# dependencies
node_modules/

# secrets & env
.env

# build output
dist/
build/

# OS / editor junk
.DS_Store
*.log
🔑
Never commit secrets

API keys, passwords and .env files must be in .gitignore before your first commit. Once a secret is pushed to GitHub it's in the history forever — treat it as leaked and rotate it.

09

Resolving merge conflicts

A conflict happens when two branches changed the same lines and Git can't decide which to keep. It's not an error — it just needs a human to choose. It looks like this:

a conflict in a filetext
<<<<<<< HEAD
<h1>Welcome to my site</h1>        ← your version (current branch)
=======
<h1>Hello and welcome!</h1>        ← their version (incoming branch)
>>>>>>> feature-login

To resolve it: edit the file to keep what you want, delete the <<< / === / >>> markers, then stage and commit:

finish the mergebash
git add index.html      # mark the conflict as resolved
git commit              # complete the merge
💡
Your editor helps

VS Code highlights conflicts and shows buttons — Accept Current, Accept Incoming, Accept Both — so you rarely edit the markers by hand.

MINI PROJECT

Put a project on GitHub

Take any project you've built — one of the mini projects from the HTML/CSS/JS guides — and publish it to GitHub from scratch.

🎯
Goal

Go from a plain folder to a live GitHub repo with your code in it, using everything from this guide.

🪜 Steps

1
On github.com, click New repository, name it, and create it empty (no README).
2
In your project folder, run the commands below.
3
Refresh the GitHub page — your files are there!
publish.shbash
cd my-project

git init                          # 1. start tracking
echo "node_modules/" > .gitignore  # 2. ignore junk (if any)

git add .                         # 3. stage everything
git commit -m "Initial commit"    # 4. first snapshot

git branch -M main                # 5. name the branch main
git remote add origin https://github.com/you/my-project.git   # 6. link GitHub
git push -u origin main           # 7. push it up 🚀
✓
From now on, the cycle is just: git add . → git commit -m "…" → git push.
✓
Add a README.md describing your project — it shows on the repo's front page.
✓
GitHub repos double as a portfolio — share the link on your CV.
🎓
