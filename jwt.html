JWT — JSON Web Tokens

A JWT is a signed, self-contained token that proves who a user is. After login, the server hands one out; the user sends it back on every request, and the server trusts it because of its signature. It's the modern way to keep APIs logged in.

🎟️
A festival wristband

At the gate you show ID once and get a tamper-proof wristband. Inside, staff just glance at the band — they don't re-check your ID. A JWT is that wristband: proof you already authenticated, that any part of the system can verify at a glance.

This guide pairs with Authentication (login, hashing, cookies vs sessions). Here we focus on the token itself.

02

What a JWT is

JWT stands for JSON Web Token. It's a string that packs some data (your identity) plus a signature the server can check. The magic: it's tamper-evident — change one character and the signature no longer matches.

✓
Self-contained — it carries the user's info, so the server needn't store a session.
✓
Signed, not encrypted — anyone can read it, but no one can forge it without the secret.
✓
Stateless — perfect for APIs and multiple servers; just verify the signature.
⚠️
Readable ≠ secret

A JWT is only encoded (Base64), not encrypted — anyone can decode and read the payload. So never put secrets (passwords, card numbers) inside it. Store only an id and maybe a role.

03

header.payload.signature

A JWT is three Base64 chunks joined by dots. You'll recognise one instantly by those two dots.

a JWTtoken
eyJhbGciOiJIUzI1NiJ9 . eyJpZCI6IjQyIiwicm9sZSI6InVzZXIifQ . 4pq7s-Xb...
└──── HEADER ────────┘ └──────── PAYLOAD ──────────────┘ └─ SIGNATURE ─┘
PartHolds
HeaderThe token type & signing algorithm, e.g. { "alg": "HS256" }
PayloadThe claims — your data: { "id": "42", "role": "user", "exp": ... }
SignatureHeader + payload, hashed with your secret key. This is what makes it forge-proof.
💡
See inside one

Paste any token into jwt.io to see its decoded header and payload. You'll see the data in plain sight — proof it's encoded, not encrypted.

04

How JWT auth works

The whole lifecycle, end to end.

the token lifecycletext
1. POST /login (email + password)
2. Server checks credentials, then SIGNS a token   ──▶  sends it back
3. Client stores the token
4. For each protected request, client sends:
       Authorization: Bearer <token>
5. Server VERIFIES the signature → trusts the payload → allows the request
   (no database lookup needed — the token IS the proof)
📘
The "Bearer" convention

Tokens travel in the Authorization header as Bearer <token>. "Bearer" just means "whoever holds this token is granted access" — so guard it like a key.

05

Signing a token

Use the jsonwebtoken library. jwt.sign() takes your payload, a secret, and options like expiry, and returns the token string.

installbash
npm install jsonwebtoken
sign.jsjavascript
const jwt = require("jsonwebtoken");

// after a successful login:
const token = jwt.sign(
  { id: user._id, role: user.role },   // the payload (keep it small!)
  process.env.JWT_SECRET,              // the secret — from .env, never hard-coded
  { expiresIn: "1h" }                  // auto-expires in an hour
);

res.json({ token });   // send it to the client
🔑
The secret is everything

Anyone with JWT_SECRET can forge valid tokens for any user. Make it long and random, keep it in .env, and never commit it.

06

Verifying a token

On each protected request, jwt.verify() checks the signature and expiry. If valid, it returns the payload; if not, it throws — so wrap it in try/catch.

verify.jsjavascript
try {
  const payload = jwt.verify(token, process.env.JWT_SECRET);
  console.log(payload.id);    // "42" — we trust this now
} catch (err) {
  // thrown if the signature is wrong OR the token has expired
  console.log("Invalid or expired token");
}
📘
No database needed

Verification is pure maths on the token + secret — that's why JWT scales so well. The server doesn't look anything up; the valid signature is the proof.

07

Auth middleware & protecting routes

Wrap verification in an Express middleware so any route can be protected by simply adding it.

auth.middleware.jsjavascript
const jwt = require("jsonwebtoken");

function auth(req, res, next) {
  // header looks like: "Authorization: Bearer eyJhbGci..."
  const token = req.headers.authorization?.split(" ")[1];
  if (!token) return res.status(401).json({ error: "No token" });

  try {
    req.user = jwt.verify(token, process.env.JWT_SECRET);  // attach the payload
    next();                                                 // let it through
  } catch {
    res.status(401).json({ error: "Invalid or expired token" });
  }
}

module.exports = auth;

// usage — protect any route:
app.get("/profile", auth, (req, res) => {
  res.json({ id: req.user.id });   // available because middleware set it
});
08

Where to store tokens

Once the client has a token, where does it keep it? Two options, each with a trade-off — this is one of the most-debated questions in web auth.

💾

localStorage

Easy: save it, add it to the Authorization header yourself. But any script can read it — risky if your site has an XSS hole.

🍪

HttpOnly cookie

The browser sends it automatically and JS can't read it (safer from XSS). Needs Secure + SameSite to guard against CSRF.

client.js (localStorage approach)javascript
// after login — save the token
localStorage.setItem("token", data.token);

// on each request — send it in the header
fetch("/api/profile", {
  headers: { Authorization: `Bearer ${localStorage.getItem("token")}` }
});
💡
Rule of thumb

For maximum safety, store tokens in an HttpOnly cookie. localStorage is simpler and fine for learning projects — just be aware of the XSS trade-off. (More on cookies vs sessions in the Auth guide.)

09

Token expiry & refresh

Tokens should expire — a stolen one shouldn't work forever. But short expiry means users get logged out often. The fix: two tokens.

TokenLivesJob
Access tokenShort (~15 min)Sent on every request; if stolen, it expires fast.
Refresh tokenLong (days)Stored securely; used only to get a new access token.
refresh.jsjavascript
const accessToken  = jwt.sign({ id }, SECRET, { expiresIn: "15m" });
const refreshToken = jwt.sign({ id }, REFRESH_SECRET, { expiresIn: "7d" });

// when the access token expires, the client calls /refresh
// with the refresh token to get a fresh access token —
// no need to log in again.
⚠️
Logout is the catch

Because JWTs are stateless, you can't truly "delete" one server-side — it's valid until it expires. Real logout means deleting it client-side and/or keeping a small server-side denylist of revoked refresh tokens.

MINI PROJECT

A token-protected route

A tiny but complete example: a /login that issues a JWT, and a /secret route only reachable with a valid one. Run it, then test with curl.

⚙️ server.js

server.jsjavascript
require("dotenv/config");
const express = require("express");
const jwt = require("jsonwebtoken");

const app = express();
app.use(express.json());
const SECRET = process.env.JWT_SECRET;

// LOGIN — (skipping password checks for brevity; see the Auth guide)
app.post("/login", (req, res) => {
  const token = jwt.sign({ id: 42, name: "Ada" }, SECRET, { expiresIn: "1h" });
  res.json({ token });
});

// MIDDLEWARE — verify the token
function auth(req, res, next) {
  const token = req.headers.authorization?.split(" ")[1];
  if (!token) return res.status(401).json({ error: "No token" });
  try {
    req.user = jwt.verify(token, SECRET);
    next();
  } catch {
    res.status(401).json({ error: "Invalid token" });
  }
}

// PROTECTED route
app.get("/secret", auth, (req, res) => {
  res.json({ message: `Welcome, ${req.user.name}! 🎉` });
});

app.listen(3000, () => console.log("http://localhost:3000"));

🧪 Try it

test.shbash
# 1. log in → get a token
curl -X POST http://localhost:3000/login

# 2. call /secret WITHOUT a token → 401
curl http://localhost:3000/secret

# 3. call /secret WITH the token → success
curl http://localhost:3000/secret \
  -H "Authorization: Bearer PASTE_TOKEN_HERE"
✓
/login signs a token; the client keeps it.
✓
No token → 401. Valid token → the protected data.
✓
The server stored nothing — the token alone proved the identity.
