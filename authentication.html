Authentication & authorization

How does an app know who you are, and what you're allowed to do? That's authentication and authorization — the gate and the guard of every real application.

🛂
Airport security

Authentication is showing your passport at check-in — proving you are who you say. Authorization is the gate agent checking your boarding pass — deciding which flight (resources) you're allowed onto.

This guide uses Node, Express and MongoDB, and pairs with the JWT guide for the token details.

02

Authentication vs authorization

Two words that sound alike and are constantly confused. They happen in order: first we authenticate, then we authorize.

🔑

Authentication (authn)

"Who are you?" Verifying identity — usually email + password, or a Google login. Happens once, at sign-in.

🛡️

Authorization (authz)

"What may you do?" Checking permissions — is this user an admin? do they own this post? Happens on every protected action.

 AuthenticationAuthorization
QuestionWho are you?What are you allowed to do?
ExampleLogging in"Only admins can delete"
Fails with401 Unauthorized403 Forbidden
💡
401 vs 403, again

401 = not logged in (failed authentication). 403 = logged in but not allowed (failed authorization). The status codes guide makes the same point.

03

Hashing passwords with bcrypt

Golden rule: never store passwords as plain text. Instead store a hash — a scrambled, one-way version. Even if your database leaks, the real passwords stay hidden.

hash.jsjavascript
const bcrypt = require("bcrypt");   // npm install bcrypt

// when REGISTERING — hash the password before saving
const hash = await bcrypt.hash("mypassword123", 10);   // 10 = salt rounds
// store `hash` in the database, NOT the raw password

// when LOGGING IN — compare the attempt to the stored hash
const ok = await bcrypt.compare("mypassword123", hash);  // true / false
📘
Why hashing (not encryption)?

Hashing is one-way — you can't reverse it back to the password. You never need to; at login you just hash the attempt and compare. The "salt" makes every hash unique, so identical passwords don't look the same.

🚫
Never do this

Don't store plain passwords, don't email passwords, and don't roll your own crypto. Use a vetted library (bcrypt, argon2) — always.

04

Cookies & sessions (vs tokens)

Logging in is the easy part. The hard part is staying logged in — HTTP is stateless, so the server forgets you after every request. There are two classic ways to keep you signed in.

🍪 What is a cookie?

A cookie is a small piece of data the server asks the browser to store. The browser then automatically sends it back on every request to that site — so the server can recognise you.

a cookiehttp
# server sets it on login:
Set-Cookie: sessionId=abc123; HttpOnly; Secure; SameSite=Strict

# browser sends it back automatically on every request:
Cookie: sessionId=abc123

🗂️ What is a session?

A session is data the server keeps about a logged-in user. The cookie holds only a random session ID; the server looks that ID up in its store (memory, Redis, a database) to find who you are.

session flowtext
1. You log in            ──▶  server creates a session, stores it
2. Server replies        ──▶  Set-Cookie: sessionId=abc123
3. Next request          ──▶  browser sends Cookie: sessionId=abc123
4. Server looks up abc123 ──▶ "ah, this is Ada" → allows the request

⚖️ Sessions vs tokens (JWT)

The modern alternative is a token (a JWT): instead of the server remembering you, it hands you a signed token that contains your identity. You send it back each time and the server just verifies the signature — nothing to store.

 Sessions + cookiesTokens (JWT)
Who remembers?The server (has state)Nobody — the token holds it (stateless)
Stored whereSession ID in a cookieThe whole token (cookie or header)
Scales across serversNeeds a shared storeEasily — just verify the signature
Revoke instantlyEasy (delete the session)Harder (wait for expiry)
Great forTraditional web appsAPIs, mobile, microservices
💡
Cookie flags that matter

HttpOnly (JavaScript can't read it — blocks XSS theft), Secure (HTTPS only), and SameSite (limits cross-site sending — blocks CSRF). A login cookie should have all three.

05

Register & login flow

Putting hashing and tokens together — the two endpoints every app needs. (Full JWT details are in the JWT guide.)

auth.routes.jsjavascript
const bcrypt = require("bcrypt");
const jwt = require("jsonwebtoken");

// REGISTER
app.post("/register", async (req, res) => {
  const { email, password } = req.body;

  const hash = await bcrypt.hash(password, 10);     // hash it
  const user = await User.create({ email, password: hash });

  res.status(201).json({ id: user._id, email });    // never send the hash
});

// LOGIN
app.post("/login", async (req, res) => {
  const { email, password } = req.body;

  const user = await User.findOne({ email });
  if (!user) return res.status(401).json({ error: "Invalid credentials" });

  const ok = await bcrypt.compare(password, user.password);
  if (!ok)  return res.status(401).json({ error: "Invalid credentials" });

  // issue a signed token (see the JWT guide)
  const token = jwt.sign({ id: user._id }, process.env.JWT_SECRET, { expiresIn: "1h" });
  res.json({ token });
});
⚠️
Same error for both

If the email doesn't exist or the password is wrong, return the same "Invalid credentials" message. Telling attackers "that email exists" leaks information.

06

Protecting routes

An auth middleware guards private routes — it checks the token (or session) before the handler runs, and attaches the user so the handler knows who's asking.

protect.jsjavascript
function protect(req, res, next) {
  const token = req.headers.authorization?.split(" ")[1];   // "Bearer xxx"
  if (!token) return res.status(401).json({ error: "Not logged in" });

  try {
    req.user = jwt.verify(token, process.env.JWT_SECRET);   // adds the user
    next();                                                 // allow through
  } catch {
    res.status(401).json({ error: "Invalid token" });
  }
}

// AUTHORIZATION — a role check on top of authentication
function adminOnly(req, res, next) {
  if (req.user.role !== "admin") return res.status(403).json({ error: "Forbidden" });
  next();
}

app.get("/profile", protect, (req, res) => res.json(req.user));
app.delete("/users/:id", protect, adminOnly, deleteUser);   // both checks
📘
authn then authz

protect answers "are you logged in?" (authentication). adminOnly answers "are you allowed?" (authorization). Chaining them is exactly the two-step gate from the start of this guide.

07

OAuth — "Sign in with Google"

OAuth lets users log in with an account they already have (Google, GitHub) instead of making a new password. Your app never sees their password — Google vouches for them.

the OAuth dancetext
1. User clicks "Sign in with Google"
2. Your app redirects them to Google's login
3. They approve → Google redirects back with a code
4. Your server swaps the code for the user's profile (email, name)
5. You create/find that user and issue YOUR own token
💡
Don't build it from scratch

Use a library like Passport.js (passport-google-oauth20) — it handles the redirects and token exchange. You register your app in the Google Cloud Console to get a client ID & secret.

08

Security best practices

A short checklist that prevents the most common auth mistakes.

✓
Always hash passwords with bcrypt/argon2 — never plain text, never reversible encryption.
✓
Always HTTPS in production — passwords and tokens must never travel in plain text.
✓
Keep secrets in .env — the JWT secret, OAuth keys — and .gitignore them.
✓
Set token expiry — short-lived access tokens, optional refresh tokens (see JWT).
✓
HttpOnly + Secure + SameSite cookies if you store tokens in cookies.
✓
Validate & rate-limit login attempts to slow brute-force attacks.
✓
Generic error messages — don't reveal whether the email or the password was wrong.
MINI PROJECT

A complete auth flow

Add real authentication to the Tasks API: users register, log in, and only see their own tasks. It combines hashing, tokens, protected routes and authorization.

🎯
The pieces

A User model · /register & /login · a protect middleware · tasks scoped to req.user.id.

⚙️ Tasks, scoped to the logged-in user

tasks.routes.jsjavascript
// every task route is behind `protect` — must be logged in
app.get("/tasks", protect, async (req, res) => {
  // only THIS user's tasks (authorization by ownership)
  const tasks = await Task.find({ owner: req.user.id });
  res.json(tasks);
});

app.post("/tasks", protect, async (req, res) => {
  const task = await Task.create({
    title: req.body.title,
    owner: req.user.id,        // stamp it with the owner
  });
  res.status(201).json(task);
});

app.delete("/tasks/:id", protect, async (req, res) => {
  const task = await Task.findById(req.params.id);
  if (!task) return res.status(404).json({ error: "Not found" });

  // can't delete someone else's task!
  if (task.owner.toString() !== req.user.id) {
    return res.status(403).json({ error: "Forbidden" });
  }
  await task.deleteOne();
  res.status(204).end();
});
✓
Authentication: protect blocks anyone without a valid token (401).
✓
Authorization: the ownership check stops users touching others' tasks (403).
✓
The owner field links each task to a user — a Mongoose relationship.
