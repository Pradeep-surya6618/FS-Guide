Express.js

Express is the most popular framework for building servers in Node. It turns the tedious raw-http work from the last guide into a few clean, readable lines — routes, JSON, middleware, all built in.

🛣️
From dirt track to highway

The bare Node server made you check req.url and req.method by hand for every route. Express gives you app.get(...), app.post(...) — a clean road system for requests.

By the end you'll build a Tasks REST API with full CRUD. In the MongoDB guide you'll give it a real database; in the Full-Stack guide a React front-end talks to it.

02

Setting up Express

Create a project, install Express, and write the smallest possible server.

setupbash
npm init -y
npm install express
app.jsjavascript
const express = require("express");
const app = express();

app.use(express.json());   // lets the app read JSON request bodies

app.get("/", (req, res) => {
  res.send("Hello from Express!");
});

const PORT = 3000;
app.listen(PORT, () => console.log(`Running on http://localhost:${PORT}`));

Run node app.js and visit http://localhost:3000. Compare it to the raw Node server — same result, far less code.

03

Routes & HTTP methods

A route ties an HTTP method + URL path to a handler function. Express has one method per HTTP verb — straight from the REST guide.

routes.jsjavascript
app.get("/users", (req, res) => { /* read all */ });
app.get("/users/:id", (req, res) => { /* read one */ });
app.post("/users", (req, res) => { /* create */ });
app.put("/users/:id", (req, res) => { /* update */ });
app.delete("/users/:id", (req, res) => { /* delete */ });
ExpressDoes (CRUD)
app.get()Read
app.post()Create
app.put() / app.patch()Update
app.delete()Delete
04

req & res objects

Every handler gets two arguments: req (the incoming request — what the client sent) and res (the response — how you reply).

📥

req — read from it

req.params (URL params), req.query (?query), req.body (JSON sent), req.headers.

📤

res — reply with it

res.json(data), res.send(text), res.status(code), res.sendFile(...).

handler.jsjavascript
app.post("/users", (req, res) => {
  const newUser = req.body;            // the JSON the client sent

  // ...save it...

  res.status(201).json({              // reply with a 201 + JSON
    message: "Created",
    user: newUser
  });
});
⚠️
Don't forget express.json()

req.body is undefined until you add app.use(express.json()). That middleware parses incoming JSON into a JS object for you.

05

Route & query params

Two ways the client passes data in the URL — and how Express reads each.

params.jsjavascript
// ROUTE param — part of the path, marked with :name
// GET /users/42
app.get("/users/:id", (req, res) => {
  res.json({ id: req.params.id });     // "42"
});

// QUERY params — after the ? , for options/filters
// GET /search?term=react&page=2
app.get("/search", (req, res) => {
  const { term, page } = req.query;    // "react", "2"
  res.json({ term, page });
});
💡
Which to use?

:id route params identify a specific resource. Query params (?sort=...) tweak a request — filtering, sorting, paging. Same distinction as in the API guide.

06

Middleware

Middleware is a function that runs between the request arriving and your route handler. It can inspect or change req/res, then call next() to pass control along.

middleware.jsjavascript
// a custom logger — runs on EVERY request
app.use((req, res, next) => {
  console.log(`${req.method} ${req.url}`);
  next();   // hand off to the next middleware / route
});

// built-in & third-party middleware
app.use(express.json());          // parse JSON bodies
app.use(express.static("public")); // serve files from /public
// app.use(cors());               // allow cross-origin requests (npm i cors)
📘
A pipeline of steps

Think of middleware as a relay: each runs in order, doing one job — logging, parsing, auth checks — then calls next(). Forget next() and the request hangs forever.

07

Error handling

Return the right status code for each failure, and use a central error handler (a middleware with four arguments) to catch everything else.

errors.jsjavascript
app.get("/users/:id", (req, res) => {
  const user = findUser(req.params.id);
  if (!user) {
    return res.status(404).json({ error: "User not found" });
  }
  res.json(user);
});

// 404 for any unmatched route (put AFTER all routes)
app.use((req, res) => res.status(404).json({ error: "Route not found" }));

// central error handler — 4 args tells Express it's for errors
app.use((err, req, res, next) => {
  console.error(err);
  res.status(500).json({ error: "Something went wrong" });
});
08

Controllers & structure

As an app grows, don't pile every route into app.js. Split it: routers group related routes, controllers hold the logic.

a tidy projectfolders
my-api/
├── app.js                 # creates the app, mounts routers
├── routes/
│   └── tasks.routes.js    # defines the URLs
├── controllers/
│   └── tasks.controller.js # the logic for each route
└── models/                # data models (MongoDB guide)
routes/tasks.routes.jsjavascript
const express = require("express");
const router = express.Router();
const { getTasks, addTask } = require("../controllers/tasks.controller");

router.get("/", getTasks);
router.post("/", addTask);

module.exports = router;

// in app.js:
app.use("/tasks", require("./routes/tasks.routes"));   // mounts at /tasks
💡
Why bother

Separating routes from logic keeps files small and testable, and means two people can work on different features without colliding. It's the standard layout for real Express apps.

MINI PROJECT

Tasks REST API

Build a complete CRUD API for a to-do list. It stores tasks in an array for now — in the MongoDB guide you'll swap that array for a real database without changing the routes.

🎯
The endpoints

GET /tasks · POST /tasks · PUT /tasks/:id · DELETE /tasks/:id — the four CRUD operations over HTTP.

⚙️ app.js

app.jsjavascript
const express = require("express");
const app = express();
app.use(express.json());

let tasks = [{ id: 1, title: "Learn Express", done: false }];
let nextId = 2;

// READ all
app.get("/tasks", (req, res) => {
  res.json(tasks);
});

// CREATE
app.post("/tasks", (req, res) => {
  const { title } = req.body;
  if (!title) return res.status(400).json({ error: "Title is required" });

  const task = { id: nextId++, title, done: false };
  tasks.push(task);
  res.status(201).json(task);
});

// UPDATE (toggle done / change title)
app.put("/tasks/:id", (req, res) => {
  const task = tasks.find((t) => t.id === Number(req.params.id));
  if (!task) return res.status(404).json({ error: "Task not found" });

  if (req.body.title !== undefined) task.title = req.body.title;
  if (req.body.done  !== undefined) task.done  = req.body.done;
  res.json(task);
});

// DELETE
app.delete("/tasks/:id", (req, res) => {
  tasks = tasks.filter((t) => t.id !== Number(req.params.id));
  res.status(204).end();   // 204 = success, no content
});

app.listen(3000, () => console.log("API on http://localhost:3000"));

▶️ How to run it

The whole app is one app.js file plus Express. In an empty folder, run these once, then start the server:

terminalbash
npm init -y            # create package.json
npm install express    # download Express into node_modules
node app.js            # start the server
# → API on http://localhost:3000
💡
No "type": "module" needed here

This file uses require(...) (CommonJS), so the default package.json from npm init works as-is — don't add "type": "module" (that's only for the import style). For auto-restart on save, install nodemon (npm install -D nodemon) and run npx nodemon app.js.

🚫
Common trip-ups

• Cannot find module 'express' → you skipped npm install express. • EADDRINUSE :3000 → port 3000 is already taken (another server is running) — stop it, or change the port in app.listen. • Leave this terminal open while the server runs; Ctrl+C stops it.

🧪 Try it

Only GET works by typing a URL in the browser — open http://localhost:3000/tasks and you'll see the JSON. POST / PUT / DELETE need a request tool: Thunder Client (a VS Code extension), Postman, or curl from a second terminal:

test.shbash
# list tasks
curl http://localhost:3000/tasks

# create one
curl -X POST http://localhost:3000/tasks \
  -H "Content-Type: application/json" \
  -d '{"title":"Build an API"}'

# delete task 1
curl -X DELETE http://localhost:3000/tasks/1

🌐 Or test it right in the browser

⚠️
The address bar only does GET

Typing a URL is always a GET — that's why /tasks shows all tasks. You can't POST/PUT/DELETE from the URL bar. To do those in the browser, use the DevTools Console.

Open http://localhost:3000/tasks in the browser (a page served by your API), press F12 → Console, and paste these. Using relative URLs (/tasks) keeps it same-origin, so there's no CORS block:

DevTools Consolejavascript
// READ all
fetch("/tasks").then((r) => r.json()).then(console.log);

// CREATE
fetch("/tasks", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ title: "New task from the browser" }),
}).then((r) => r.json()).then(console.log);

// UPDATE task 1 (mark it done)
fetch("/tasks/1", {
  method: "PUT",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ done: true }),
}).then((r) => r.json()).then(console.log);

// DELETE task 1
fetch("/tasks/1", { method: "DELETE" }).then((r) => console.log("status:", r.status));
💡
Seeing the change

After a CREATE/UPDATE/DELETE, run the READ line again (or just refresh /tasks in the tab) to see the updated list. Even easier for beginners: install the Thunder Client extension in VS Code — it's a point-and-click way to pick the method, URL and JSON body, no console needed.

✓
Each route returns the right status: 201 created, 400 bad input, 404 missing, 204 deleted.
✓
req.body carries the JSON; req.params.id picks the task.
✓
The data is in memory — it resets when the server restarts. Persistence is next.
🎓
