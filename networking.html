FUNDAMENTALS · 01

Networking basics

Every time you load a page, dozens of messages fly across the internet in well under a second. This guide demystifies that journey — what a client and server are, how a domain name becomes a connection, and what travels back and forth.

📮
The postal analogy

Sending a request is like posting a letter. You need an address (the IP), a post office to look it up by name (DNS), an envelope with sender/recipient info (headers), and the letter inside (the body). The reply comes back the same way.

You don't need to memorise protocols — just build a clear mental model. It makes the API, Node and Full-Stack guides click into place.

02

Client & server

Almost all web traffic is a conversation between two roles: a client that asks, and a server that answers.

💻

Client

The one that requests. Usually your browser or phone app. It sends a request and waits for the reply.

🖥️

Server

A computer that listens for requests and responds — with a web page, JSON data, an image, anything.

the conversationtext
CLIENT  ──────── "GET me the home page" ───────▶  SERVER
   │                                                 │
   ◀──────── "200 OK, here's the HTML" ──────────────┘
📘
Servers only respond

A server can't push to a client out of the blue — the client always starts the conversation by making a request. (Live updates use special tricks like WebSockets, built on top of this.)

03

IP addresses & DNS

Every device on a network has an IP address — its numeric location. But nobody types 142.250.183.110; we type google.com. DNS is the system that translates one into the other.

🔢

IP address

A device's address, like 142.250.183.110 (IPv4) or a longer IPv6. Servers have one so requests can find them.

📖

DNS

The internet's phonebook: it maps a friendly domain (google.com) to its IP address.

a DNS lookuptext
You type:     github.com
                  │
DNS lookup:   "what's the IP for github.com?"  ──▶  140.82.121.4
                  │
Browser:      connects to 140.82.121.4 and makes the request
💡
A domain has parts

In api.github.com: com is the top-level domain, github is the domain, and api is a subdomain. DNS results are cached, so repeat visits skip the lookup.

04

The request/response cycle

Here's everything that happens between pressing Enter on a URL and seeing the page — in order.

1
DNS lookup — turn the domain into an IP address.
2
Connect — open a connection to that IP (a TCP handshake; HTTPS adds a TLS handshake).
3
Request — the browser sends an HTTP request (method + path + headers).
4
Server works — runs code, maybe queries a database, builds a response.
5
Response — comes back with a status code, headers and a body (HTML/JSON).
6
Render — the browser paints the page and fetches more assets (CSS, JS, images) the same way.
a raw HTTP requesthttp
GET /users/torvalds HTTP/1.1
Host: api.github.com
Accept: application/json

# ...and the response:
HTTP/1.1 200 OK
Content-Type: application/json

{ "login": "torvalds", "public_repos": 6 }
05

HTTP methods

Every request has a method — a verb that says what you want to do with the resource.

MethodMeans
GETRead / fetch data (the default when you visit a page)
POSTSend data to create something (form submit, sign up)
PUT / PATCHUpdate an existing thing
DELETERemove a thing
💡
More detail in the API guide

These methods are the heart of REST APIs — how they map to CRUD and how to call them with fetch is covered in the APIs & REST guide.

06

Headers & body

Both requests and responses have two parts: headers (info about the message) and an optional body (the actual data).

🏷️

Headers

Key–value metadata: content type, length, auth tokens, caching rules. Like the writing on an envelope.

📦

Body

The payload — the HTML page, the JSON data, the uploaded file. GET requests usually have no body.

Common headerSays
Content-TypeWhat format the body is — application/json, text/html
AuthorizationWho you are — a token or key (see JWT)
AcceptWhat format you'd like back
Cache-ControlHow long this can be cached
User-AgentWhat browser/app is making the request
07

Ports & localhost

An IP address gets you to the right computer; a port gets you to the right program on it. One server can run many services, each on its own port.

a URL with a porturl
http://localhost:3000/api
       └───┬───┘ └─┬┘ └─┬─┘
        host     port  path

localhost = "this computer" (the IP 127.0.0.1)
PortUsually
80HTTP (the default — hidden in normal URLs)
443HTTPS (the secure default)
3000 / 5173 / 8080Dev servers (Node/Express, Vite, etc.)
27017MongoDB
💡
You'll see this constantly

When you run a Node/Express server it prints something like listening on http://localhost:3000 — that's your machine acting as the server, reachable on port 3000.

08

HTTPS & TLS

HTTPS is just HTTP with encryption. The S stands for Secure — it's the padlock 🔒 in your address bar. The encryption layer is called TLS (you may hear the older name SSL).

🔓

HTTP

Plain text. Anyone between you and the server (cafe wifi, your ISP) could read it. Fine for nothing sensitive.

🔒

HTTPS

Encrypted with TLS, so only the server can read your data. Required for logins, payments — really, everything today.

HTTPS does two things: it encrypts the traffic, and a certificate proves the server really is who it claims to be. The browser checks the certificate automatically and shows the padlock when all's well.

⚠️
Never send secrets over plain HTTP

Passwords, tokens and card details must go over HTTPS. Modern browsers warn on http:// pages with password fields for exactly this reason.

09

Latency & caching

Data takes time to travel. Latency is that delay; caching is how we avoid paying it twice.

⏱️

Latency

The round-trip time for a request — measured in milliseconds. Distance, network quality and server speed all add to it.

🗄️

Caching

Storing a copy of a response so the next request is instant — in the browser, or on a CDN close to the user.

📘
CDNs bring data closer

A CDN (Content Delivery Network) keeps copies of files on servers around the world, so a user in India loads from a nearby server instead of one across the planet — slashing latency. It's why a 304 Not Modified response (see status codes) is so fast.

MINI PROJECT

Trace a real page load

No code for this one — you'll use the tool every developer lives in: the browser's DevTools Network tab. Watch the request/response cycle happen for real.

🎯
Goal

See the actual requests, methods, status codes, headers and timings behind a page you visit every day.

🪜 Steps

1
Open any site, press F12, and click the Network tab.
2
Reload the page. Watch the list of requests fill up — HTML first, then CSS, JS, images, fonts.
3
Click the first request. Find its Method (GET), Status (200), and the Headers tab.
4
Spot the Content-Type header and, on a data request, switch to Response to see the JSON.
5
Check the Timing — DNS lookup, connecting, waiting (TTFB), downloading. That's everything from this guide, visualised.

🧪 Bonus: a request from the terminal

See a raw response with curl — the request and headers, no browser involved:

terminalbash
# -i shows the status line and headers, then the body
curl -i https://api.github.com/users/torvalds

# just the headers (a HEAD request)
curl -I https://github.com
