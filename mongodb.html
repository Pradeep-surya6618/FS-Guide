DATABASE · 01

MongoDB

Your Express API works, but forgets everything on restart because the data lives in a variable. A database stores it permanently. MongoDB is the most popular database for JavaScript apps — it stores data as JSON-like documents.

🗄️
A filing cabinet for data

A collection is a drawer (e.g. "tasks"), and each document is a folder inside it (one task). Unlike a spreadsheet, every document can have a slightly different shape.

You'll use Mongoose, the library that connects Node to MongoDB and adds structure. It completes the MERN stack (Mongo, Express, React, Node) — see the Full-Stack guide.

02

What is MongoDB?

MongoDB is a NoSQL database. Instead of tables with fixed rows and columns (like SQL), it stores flexible documents that look just like JavaScript objects.

📊

SQL (e.g. MySQL)

Rigid tables, rows & columns. Every row has the same fixed shape. You query with the SQL language.

🍃

MongoDB (NoSQL)

Flexible JSON-like documents. Great fit for JavaScript — the data is objects. You query with JS objects.

📘
It feels like JavaScript

A MongoDB document is basically a JS object that's saved to disk. No translating between "database rows" and "app objects" — they're the same shape. That's why it pairs so naturally with Node.

03

MongoDB Atlas setup

Atlas is MongoDB's free cloud database — no install needed. A few clicks gives you a database and a connection string.

1
Sign up at mongodb.com/atlas and create a free (M0) cluster.
2
Create a database user (a username & password for your app).
3
Under Network Access, allow your IP (or 0.0.0.0/0 for learning).
4
Click Connect → Drivers and copy the connection string.
.envtext
# paste your string here, with your real password
MONGODB_URI=mongodb+srv://user:password@cluster0.xxxxx.mongodb.net/myapp
🔑
Keep the URI secret

The connection string contains your password — keep it in .env and .gitignore it (see Node · env variables). Never paste it into code that gets pushed to GitHub.

04

Documents & collections

Two words to know. A document is one record (JSON-like). A collection is a group of documents of the same type.

a documentjson
// one document in the "tasks" collection
{
  "_id": "6512f1a4c3...",     // MongoDB adds a unique id automatically
  "title": "Learn MongoDB",
  "done": false,
  "tags": ["backend", "database"],
  "createdAt": "2026-06-30T10:00:00Z"
}
MongoDBSpreadsheet analogy
DatabaseThe whole workbook
CollectionA sheet (e.g. "tasks")
DocumentA row
FieldA cell / column value
💡
The _id field

Every document gets a unique _id automatically (an ObjectId). You use it to fetch, update or delete that exact document.

05

Connecting with Mongoose

Mongoose is the library that connects your Node app to MongoDB and adds helpful structure (schemas, validation, easy queries). Install it, then connect once when the app starts.

installbash
npm install mongoose dotenv
db.jsjavascript
require("dotenv/config");
const mongoose = require("mongoose");

async function connectDB() {
  try {
    await mongoose.connect(process.env.MONGODB_URI);
    console.log("✅ MongoDB connected");
  } catch (err) {
    console.error("❌ DB connection failed:", err.message);
    process.exit(1);
  }
}

module.exports = connectDB;
06

Schemas & models

A schema defines the shape of your documents (which fields, what types, what's required). A model is the tool built from that schema — you use it to read and write data.

models/Task.jsjavascript
const mongoose = require("mongoose");

const taskSchema = new mongoose.Schema(
  {
    title: { type: String, required: true },
    done:  { type: Boolean, default: false },
    tags:  [String],
  },
  { timestamps: true }   // auto-adds createdAt & updatedAt
);

// "Task" → Mongoose makes a "tasks" collection
module.exports = mongoose.model("Task", taskSchema);
📘
Schema = a friendly guardrail

MongoDB itself is schema-less, but Mongoose schemas catch mistakes early — a missing title, a number where text belongs — before bad data reaches the database.

07

Create, Read, Update, Delete

The four operations, all on the model. Each returns a promise, so use await.

crud.jsjavascript
const Task = require("./models/Task");

// CREATE
const task = await Task.create({ title: "Learn Mongoose" });

// READ
const all  = await Task.find();              // every task
const one  = await Task.findById(id);        // by _id
const done = await Task.find({ done: true }); // filtered

// UPDATE
await Task.findByIdAndUpdate(id, { done: true }, { new: true });

// DELETE
await Task.findByIdAndDelete(id);
OperationMongoose method
Createcreate() / new Model().save()
Readfind(), findById(), findOne()
UpdatefindByIdAndUpdate(), updateOne()
DeletefindByIdAndDelete(), deleteOne()
08

Querying & filtering

You query by passing a plain object that describes what to match. Operators (prefixed with $) handle ranges, lists and more.

queries.jsjavascript
// exact match
await Task.find({ done: false });

// operators: $gt, $lt, $in, $ne ...
await Task.find({ priority: { $gte: 3 } });        // priority >= 3
await Task.find({ tags: { $in: ["urgent"] } });    // has the "urgent" tag

// sort, limit, pick fields
await Task.find()
  .sort({ createdAt: -1 })   // newest first
  .limit(10)                 // only 10
  .select("title done");     // only these fields
💡
Queries are chainable

Build up a query with .sort(), .limit(), .skip() (for paging) and .select(), then await the result. It reads almost like a sentence.

09

Relationships

Real data is connected — a task belongs to a user, an order has products. In Mongoose you link documents by storing one's _id on another, then populate to fetch the full linked document.

relationships.jsjavascript
const taskSchema = new mongoose.Schema({
  title: String,
  owner: { type: mongoose.Schema.Types.ObjectId, ref: "User" },  // link
});

// store just the user's id on the task
await Task.create({ title: "Ship it", owner: user._id });

// fetch the task WITH its owner's full document
const task = await Task.findById(id).populate("owner");
console.log(task.owner.name);   // the linked User's name
📘
Reference vs embed

Two patterns: reference (store an _id, like above — good for big or shared data) or embed (nest the data directly in the document — good for small, owned data like a list of tags).

MINI PROJECT

Persist the Tasks API

Take the in-memory Express Tasks API and give it a real database. The routes barely change — you just swap the array for Mongoose calls. Now your data survives restarts.

🎯
The payoff

This is a real, production-shaped backend: Express + Mongoose + Atlas. The same code pattern powers the Full-Stack capstone.

⚙️ app.js

app.jsjavascript
require("dotenv/config");
const express = require("express");
const mongoose = require("mongoose");
const Task = require("./models/Task");

const app = express();
app.use(express.json());

// connect to MongoDB Atlas
mongoose.connect(process.env.MONGODB_URI)
  .then(() => console.log("✅ DB connected"))
  .catch((err) => console.error(err));

// READ all
app.get("/tasks", async (req, res) => {
  const tasks = await Task.find().sort({ createdAt: -1 });
  res.json(tasks);
});

// CREATE
app.post("/tasks", async (req, res) => {
  try {
    const task = await Task.create({ title: req.body.title });
    res.status(201).json(task);
  } catch (err) {
    res.status(400).json({ error: err.message });   // e.g. missing title
  }
});

// UPDATE
app.put("/tasks/:id", async (req, res) => {
  const task = await Task.findByIdAndUpdate(req.params.id, req.body, { new: true });
  if (!task) return res.status(404).json({ error: "Not found" });
  res.json(task);
});

// DELETE
app.delete("/tasks/:id", async (req, res) => {
  await Task.findByIdAndDelete(req.params.id);
  res.status(204).end();
});

app.listen(3000, () => console.log("API on http://localhost:3000"));
✓
Same routes as the Express guide — only the storage changed.
✓
Every DB call is awaited; the route handlers became async.
✓
Restart the server — your tasks are still there. 🎉
✓
Schema validation means a task with no title is rejected with a clean 400.
🎓
